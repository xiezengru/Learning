# Anthropic：给 AI Agents 编写有效工具的复习总结

来源：<https://www.anthropic.com/engineering/writing-tools-for-agents>

## 概要
- 文章的核心观点是：Agent 的能力上限常常由“你给它的工具”决定；工具设计与评估方式，直接影响 Agent 是否能稳定、低成本地完成任务。
- 从工程视角把“写工具”当作一种契约（contract）：确定性软件（工具）与非确定性智能体（Agent）之间要对输入/输出、语义和上下文质量达成一致。
- 文中给出一套可复用流程：原型验证 -> 构建评估 -> 迭代优化，并总结了多条“写工具原则”。

## 核心思想
- **工具不是 API 的简单封装**：工具需要对 Agent 的“可用性（affordances）”和有限上下文进行优化。
- **用评估驱动迭代**：把工具当作可度量的系统，通过真实任务的评估集衡量改动带来的收益，避免凭感觉优化。
- **上下文要“高信号、低噪音、可控”**：返回内容要帮助 Agent 更快做出下一步，而不是把大量无关信息塞进上下文。
- **描述本身就是行为约束**：工具描述与 schema（输入输出字段、命名、示例、边界）会显著影响 Agent 的工具调用策略。

## 主要内容
### 1) 为什么要重新思考“写工具”
- 传统软件是确定性的：相同输入 -> 相同输出。
- 对 Agent 来说，决策是非确定性的：它可能误用工具、漏调用工具，或在理解工具能力边界时出错。
- 因此写工具需要为 Agent 的不确定性做设计：让工具在“被正确理解、被高效调用、可验证结果”这几件事上更可靠。

### 2) 工具的工程流程（从原型到评估）
- **构建原型（prototype）**：尽快在本地把工具跑起来，必要时用 MCP 服务器或桌面扩展（DXT）打通测试闭环。
- **生成评估任务（evaluation tasks）**：从真实工作流抽取任务，要求提示词基于真实数据源，并且任务复杂度要足够挑战工具（可能需要多次工具调用）。
- **准备可验证响应（verifiable outcomes）**：
  - 可以用精确匹配或让模型/裁判判断是否达标；
  - 避免“过度严格”导致正确回答因标点/格式差异被拒。
- **运行评估（evaluation）**：建议用程序化的 LLM 调用评估，通常用循环结构交替执行“模型 -> 工具”。
- **记录多维指标**：不仅看准确率，也关注工具调用耗时、调用次数、token 消耗、错误率等，便于定位“为什么没用好”。
- **分析结果并迭代**：查看 Agent 的推理/反馈与原始转录（包含工具调用与返回），找出工具描述含糊、schema 误导或实现低效等问题。
- **让 Agent 帮你优化工具**：把评估转录喂给 Agent（如 Claude Code）进行批量改进，保证描述、实现与 schema 的一致性，并避免回归。

### 3) 写工具的原则（关键可操作点）
1. **选择合适的工具（不要更多就是更好）**
   - 工具数量与粒度需要匹配 Agent 的上下文限制；
   - 优先实现面向高价值工作流的“意图型工具”（例如 `search_contacts`），而不是把底层列表接口暴露给 Agent 让它“读全部再筛”。
   - 工具可以在内部合并多步操作，降低 Agent 在中间步骤上浪费上下文。
2. **Namespacing（给工具加命名空间）**
   - 当存在大量 MCP 服务器/工具时，工具名与层级要帮助 Agent 区分用途；
   - 通过前缀/后缀对服务与资源进行组织，减少“选错工具”与“传错参数”概率。
3. **返回有意义的上下文（high-signal）**
   - 工具返回要更偏“可用于下一步行动的语义信息”，少给难读的低层技术标识（如难理解的 UUID、无上下文的参数）。
   - 在需要兼容自然语言与技术标识时，可设计输出格式开关（如 `response_format`），在“精简/详细”之间切换。
4. **为 token 效率做优化**
   - 对可能产生大量文本的工具输出，使用分页、范围选择、过滤、截断，并提供合理默认值。
   - 错误信息也要可改进：不要只返回报错码，应该给出“可操作的输入修正建议”。
5. **对工具描述做 Prompt 工程**
   - 工具描述与 spec 会进入 Agent 上下文；写成“让新同事也能直接用对”的清晰说明；
   - 强调明确的参数命名、输入输出语义、边界条件与示例；
   - 利用严格的数据模型（schema）减少歧义，显著降低错误率。

## 关键知识点（复习抓手）
- **工具=契约**：为 Agent 明确输入输出语义与边界，而不是只把后端能力暴露出来。
- **评估驱动**：用真实任务评估集衡量效果，跟踪准确率之外的成本指标（tokens/耗时/调用次数/错误）。
- **任务要真实且有挑战**：避免“太简单的沙盒任务”，否则评估无法发现工具的真实薄弱点。
- **输出要高信号、低噪音**：尽量返回 Agent 下一步最需要的字段与语义信息；减少无关上下文。
- **工具粒度要贴合 Agent 的搜索/执行方式**：让 Agent 做“决策与选择”，而不是在上下文里做暴力遍历。
- **描述与 schema 也是性能的一部分**：工具描述精度提升，往往能显著降低失败率。
- **错误信息要指导修正**：良好的 error response 能把 Agent 引导到更 token-efficient 的正确策略。

## 结论 / 推荐复习点
- 结论：要让 Agent 真的“能用且好用”，核心不在模型本身，而在“工具定义 + 工具输出质量 + 评估迭代”这一整套工程系统。
- 复习建议 1：把“写工具”当成一次完整的工程闭环：原型验证 -> 真实任务评估 -> 指标分析 -> 迭代优化。
- 复习建议 2：工具设计优先级顺序可以记成：**选择合适工具 > 命名空间 > 返回高信号上下文 > token 预算与截断 > 描述与 schema 的精确度**。
- 复习建议 3：复盘时不仅看最终成功率，还要看“工具调用次数、耗时、token、失败原因”，这样才能定位是描述问题还是实现问题。
- 复习建议 4：当你要新增工具时，先写清楚输入输出与示例，并配套评估任务，避免上线后才发现 Agent 用错或用不起来。
