# How Long Contexts Fail — Summary

## 概要

这篇文章讨论了“上下文越长越好”的误解，并指出在 agent 架构中，长上下文会带来多种失败模式。作者将上下文失败分为四类：Context Poisoning、Context Distraction、Context Confusion、Context Clash，并说明这些问题为何在多轮 agent 工作流中尤为严重。

## 核心思想

- 更大的上下文窗口并不等于更好的结果。
- 过多或不当的上下文会使 agent 偏离训练信号、陷入错误认知、响应失焦，甚至内部冲突。
- 管理上下文质量比追求上下文长度更重要。

## 主要内容

### 1. Context Poisoning

- 当错误信息、幻觉或无效结论进入上下文后，会被反复引用和强化。
- 作者引用 Gemini 2.5 的案例：代理在玩宝可梦时产生幻觉，导致目标部分被“污染”，进而持续重复不可能的行动。
- 一旦目标或摘要被污染，代理可能继续沿着错误方向行动，难以恢复。

### 2. Context Distraction

- 上下文变得过长时，模型可能过度依赖上下文，而忽视自身训练内容。
- 文章指出，超过一定规模（例如 32k 以上）后，模型的正确性反而下降；对小模型而言，这个阈值更低。
- 这里的关键是，长上下文适合检索和摘要，但不一定适合多步推理和决策。

### 3. Context Confusion

- 多余或不相关的上下文也会影响模型，导致低质量答案或错误工具调用。
- 文章提到 MCP 工具描述、工具数量过多，都会让模型难以判断何时使用哪些工具。
- Berkeley 函数调用排行榜显示，模型在提供多个工具时表现更差；给小模型 46 个工具几乎必败，而只给 19 个工具时则能成功。

### 4. Context Clash

- 当上下文中出现彼此冲突的信息时，模型会陷入“内部矛盾”，其表现会显著下降。
- 引用了微软/Salesforce 的研究：将信息拆分成多轮对话后再合并，模型性能下降严重，原因是早期错误假设被保留在上下文中。
- 这对 agent 来说尤其危险，因为它们往往从多个来源（文档、工具调用、中间模型输出）聚合信息，冲突的风险更高。

## 关键知识点

- 大上下文窗口虽然技术上可行，但并不是万能解。
- 上下文的质量、相关性和一致性比长度更重要。
- agent 的失败往往来自“好信息过载”：上下文中含有很多无效、冲突或误导性内容。
- 长上下文更容易暴露出 agent 工具数量、累积历史和多轮推理的脆弱性。

## 结论

- 追求更长的上下文不应是 agent 设计的首要目标。
- 正确的做法是管理上下文：避免污染、过滤噪音、减少无关工具、解决信息冲突。
- 在复杂 agent 工作流中，控制上下文增长并保持上下文清晰比将所有内容塞进 prompt 更重要。

## 推荐复习点

- 上下文失败的四类：Poisoning、Distraction、Confusion、Clash
- 为什么长上下文会让代理变糟：不是模型能力问题，而是上下文使用方式问题
- 工具描述与工具数量如何引发 Context Confusion
- 多轮交互与上下文冲突导致 Context Clash
- 优先考虑上下文管理，而不是盲目增加上下文窗口
