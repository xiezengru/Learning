# APIs Don't Make Good MCP Tools — Summary

## 概要

本文作者认为，将现有 Web API 自动转换成 MCP 工具往往不是最佳选择。虽然可以实现，但效果通常不佳，因为 MCP 工具和传统 API 的设计目标不同。文章从工具数量、上下文窗口、工具能力以及代理自身能力四个方面进行论证。

## 核心思想

- MCP 工具应当为代理设计，而不是简单映射现有 API。
- 现有 API 往往不适合代理的调用模式和上下文限制。
- 自动化生成 MCP 工具可能带来更高的复杂度和更低的可靠性。

## 主要内容

### 1. 代理无法应对大量工具

- 许多模型在工具数量远低于 VS Code 128 个上限时就已经开始出现调用错误。
- 每个工具都占用模型上下文和工具选择空间。
- 许多 Web API 并未针对这种限制进行设计。
- 更好的 MCP 工具通常可以整合多个 API 功能，变得更灵活。

### 2. API 数据会迅速耗尽上下文窗口

- Web API 结果通常包含大量字段和记录。
- 将宽表 JSON 原样传给代理，会很快消耗 tokens。
- 数据格式也很浪费；JSON 比 CSV/TSV 更占 token。
- 直接暴露原始 API 返回值风险很大，尤其是字段数量和记录大小变动时。
- 文章指出，虽然可以通过字段投影、结果截断、JSON 转 CSV/YAML 等方式缓解，但多数服务器并未这么做。

### 3. API 未充分利用代理的独特能力

- 代理不仅能处理结构化数据，也能处理自由文本和分层指令。
- MCP 工具可以返回更适合代理下一步操作的结果，例如带建议的列表输出。
- 设计良好的 MCP 工具可以链式调用、分层处理，从而比单个 API 更有效。

### 4. 代理本身可以直接调用 API

- 现代代理（例如 Claude Code）已经能够编写和执行调用 Web API 的代码。
- 有人认为如果代理能直接安全地调用 API，则可能不需要 MCP。
- 作者不同意这个结论，但认为应该关注代理能力与安全沙箱的改进方向。

## 关键知识点

- 代理对工具数量敏感，MCP 工具应尽量精简。
- 直接将现有 API 映射为 MCP 工具会放大 API 设计与代理使用场景的不匹配。
- MCP 工具的设计应考虑：token 代价、数据格式、工具组合、后续调用建议。
- 代理能够执行代码并直接调用 API，这是一种可选备选方案，而不是 MCP 的彻底替代。

## 结论

- APIs 可以转成 MCP 工具，但通常效果不佳。
- 代理与 API 的使用语境不同，MCP 工具需要专门为代理设计。
- 最佳做法是构建适合代理的工具层，而不是简单地包装所有现有 API。

## 推荐复习点

- MCP 工具设计应优于 API 直接映射
- 工具数量与模型选择能力的限制
- 上下文窗口消耗与 JSON 代价
- 基于代理能力设计更自由的工具输出
- 直接调用 API 与 MCP 之间的权衡
