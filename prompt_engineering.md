# 提示工程

提示设计是创建提示以从语言模型引出所需回答的过程。编写结构化良好的提示是确保从语言模型获得准确优质回答的重要环节。提示工程则是通过反复调整提示并评估模型输出，不断优化交互效果的过程。

## 1 提示工程基础

### 1.1 提示的组成部分

- **任务**：模型需要完成的操作，通常以问题或说明形式出现。
- **系统指令**：在用户输入之前传递给模型的全局行为指令，用于控制风格、角色和限制条件。
- **少样本示例**：通过输入-输出对示例，引导模型理解结果格式和表达风格。
- **背景信息**：为模型提供额外上下文或数据，提升回答的准确性和相关性。

### 1.2 提示设计最佳实践

- 提供清晰而具体的说明。
- 添加少样本示例。
- 分配角色。
- 添加上下文信息。
- 使用系统说明。
- 结构化提示。
- 指示模型解释其推理。
- 分解复杂任务。
- 使用参数值进行实验。
- 进行提示迭代。

### 1.3 常用参数

- **输出词元数上限**：控制回答长度。值越低，回答越短；值越高，回答可能越长。
- **温度（temperature）**：控制采样随机性。低温度使输出更稳定、确定；高温度令结果更具创造性和多样性。推荐初始值为 1.0，Gemini 支持 0.0 至 2.0。
- **Top-P**：按概率累积筛选候选词元，直到总概率达到设定值。Top-P 越低，随机性越低；越高，随机性越高。
- **Top-K**：从概率最高的 K 个词元中抽样下一个词元。K 越小，输出越保守；K 越大，输出越随机。
- **种子（seed）**：固定种子可以提高重复请求时的结果一致性，但不保证完全确定性；更改模型或其他参数后输出仍可能变化。

> 这些参数直接影响生成结果的长度、随机性和多样性。通过实验不同参数组合，可以为具体任务找到更适合的设置。

## 3 术语说明

### 3.1 基础提示

#### 零样本提示（Zero-shot Prompt）
- 定义：直接向模型发出任务说明，不提供示例。
- 适用场景：任务明确、模型具备通用理解能力时。
- 示例：`请将下面这段文字翻译成英文：...`。

#### 少样本提示（Few-shot Prompt）
- 定义：在指令中提供少量示例，帮助模型理解任务格式和要求。
- 适用场景：任务结构复杂或容易歧义时。
- 示例：
  - `示例1：...`
  - `示例2：...`
  - `现在请按相同方式处理以下输入：...`

### 3.2 推理链类

#### 链式思考（CoT）提示（Chain-of-Thought）
- 定义：要求模型生成中间推理步骤，然后得出结论。
- 适用场景：数学推理、逻辑分析、复杂决策等。
- 示例：`先一步步思考，再给出最终答案。`

#### 链式提示（Chain Prompt）
- 定义：将任务拆成多个子步骤，按序引导模型完成。
- 适用场景：任务本身具有明确分阶段流程时。
- 示例：`第一步：分析问题；第二步：生成大纲；第三步：输出结果。`

#### 自我一致性（Self-Consistency）
- 定义：多次采样多个推理路径，然后选取最常见或最一致的答案。
- 适用场景：当单次输出可能不稳定，但多数投票可以提高准确率时。
- 示例：`先生成多次答案，再比较哪个结论最常出现。`

#### 自我反思（Reflexion）
- 定义：让模型审查自身输出并进行纠正或优化。
- 适用场景：需要提升输出质量、修正错误或生成更合规结果时。
- 示例：`请检查刚才的回答，并指出其中的错误，然后修正。`

#### 思维树（ToT，Tree of Thought）
- 定义：通过构建多个推理路径树，搜索最佳推理结果。
- 适用场景：需要深度探索多种可能路径，以避免早期局部最优。
- 示例：`分别列出几种可能的推理方向，然后评估每个方向的结果。`

#### 多模态思维链提示方法
- 定义：将多模态信息（文本、图像、表格等）与链式思考结合使用。
- 适用场景：需要结合多种输入形式进行复杂理解和推理时。
- 示例：`先分析图片内容，再结合文字描述推理结论。`

### 3.3 知识/检索增强

#### 生成知识提示（Knowledge-Enhanced Prompt）
- 定义：通过模型生成补充知识或语境信息，辅助后续任务。
- 适用场景：任务依赖背景知识或需要丰富上下文时。
- 示例：`先生成这段主题的背景知识，再基于这些知识回答问题。`

#### 检索增强生成（RAG，Retrieval-Augmented Generation）
- 定义：先检索外部知识库，再把检索结果作为上下文输入模型生成回答。
- 适用场景：需要利用实时、特定领域或大规模知识库时。
- 示例：`从文档库检索相关段落，然后根据检索结果生成摘要。`

### 3.4 工具与行为框架

#### 自动推理并使用工具（ART）
- 定义：模型在推理过程中自动决定是否调用工具，并结合工具结果完成任务。
- 适用场景：需要动态调用计算、检索、API 等外部功能时。
- 示例：`当需要计算或查询时，自动调用相应工具；否则直接继续推理。`

#### PAL（程序辅助语言模型，Program-Aided Language Model）
- 定义：通过让模型生成可执行程序代码，再运行该代码获取准确结果。
- 适用场景：涉及数值计算、数据处理、格式化等需要精确输出的任务。
- 示例：`让模型先生成 Python 代码，然后执行该代码得到答案。`

#### ReAct 框架
- 定义：将“思考（Reasoning）”与“行动（Action）”交替进行，模型可以边思考边调用工具。
- 适用场景：需要模型在推理过程中不断查询外部资源或做决定时。
- 示例：`思考：我需要查询数据库；行动：调用查询工具；思考：根据结果继续分析。`

### 3.5 自动化与优化

#### 自动提示工程师（APE）
- 定义：让模型自动生成或优化提示，成为“提示工程师”的角色。
- 适用场景：需要快速生成高质量提示或优化提示内容时。
- 示例：`请作为提示专家，帮我改写以下任务指令，使其更易于模型理解。`

#### Active-Prompt
- 定义：通过模型主动询问或扩展提示，以收集更多必要信息来完成任务。
- 适用场景：当初始提示信息不足，需要进一步澄清需求时。
- 示例：`请先问我几个问题以明确需求，然后再生成答案。`

#### 方向性刺激提示（Directional Prompting）
- 定义：在提示中加入明确方向性指令，引导模型朝期望风格或格式输出。
- 适用场景：需要控制输出风格、结构或重点时。
- 示例：`请用简洁、结构化的方式回答，避免赘述。`

## 4 使用建议

- 如果任务简单且明确，优先使用零样本或少样本提示。
- 需要推理时，使用 CoT、链式提示、自我一致性或思维树。
- 有外部知识需求时，优先采用生成知识提示或 RAG。
- 需要工具调用和多步骤行动时，选择 ART、PAL 或 ReAct。
- 需要自动优化提示时，使用 APE、Active-Prompt 或方向性刺激。
